1. Visão Geral do Projeto

O Guard Pets Intelligence é uma plataforma web de gerenciamento de resgates e adoções de animais vítimas de maus-tratos. O sistema integra uma equipe multidisciplinar formada por advogados, veterinários e especialistas em resgate tático, permitindo:

  • Recepção e rastreamento de denúncias de maus-tratos;
  • Gerenciamento do ciclo de vida de animais resgatados;
  • Processo de agendamento de adoções;
  • Cadastro e autenticação segura de agentes.
Stack principal: Frontend em HTML5 / CSS3 / JavaScript vanilla; Backend em Node.js com Express; Banco de dados MySQL; Autenticação via JWT + bcrypt.

2. Arquitetura do Sistema

Frontend (Cliente)

HTML5, CSS3 (Glassmorphism), JavaScript ES2022. Comunica com o backend via fetch API (JSON). Padrões de design implementados no lado do cliente.

Backend (Servidor)

Node.js + Express. API RESTful com endpoints para autenticação, CRUD de usuários, denúncias e ocorrências. Porta padrão: 3000.

Banco de Dados

MySQL (WAMP/localhost). Tabelas: usuarios, animais, ocorrencias, agendamentos. Script em database.sql.

Segurança

Senhas criptografadas com bcryptjs (salt 10). Sessões autenticadas com JSON Web Token (expiração em 8h). Rotas protegidas por middleware JWT.

4. Modelo de Dados (MER)

O banco de dados guard_pets é composto por quatro entidades. As formas normais 1FN, 2FN e 3FN foram aplicadas em todas as tabelas.

Entidades e Atributos

TabelaPKCampos PrincipaisRestrições Notáveis
usuarios id (AI) nome, email, cpf, senha (hash), especialidade, tipo, ativo UQ: email, cpf · NOT NULL: nome, email, senha
animais id (AI) nome, especie, raca, status (ENUM), castrado, vacinado, disponivel_adocao ENUM status · NOT NULL: nome, especie, status
ocorrencias id (AI) protocolo, tipo, localizacao, relato, nome_denunciante, status (ENUM) UQ: protocolo · ENUM status · ON UPDATE: atualizado_em
agendamentos id (AI) protocolo, nome_adotante, telefone, motivacao, animal_id (FK), status UQ: protocolo · FK → animais(id) ON DELETE SET NULL

Relacionamentos

Tabela OrigemFKTabela ReferenciadaCardinalidadeOn Delete
agendamentos animal_id animais(id) N : 1 SET NULL

Formas Normais

  • 1FN: todos os campos são atômicos — sem grupos repetidos ou arrays.
  • 2FN: todas as PKs são simples (AUTO_INCREMENT) — sem dependências parciais.
  • 3FN: nenhum atributo não-chave depende transitivamente de outro — ex.: o telefone do adotante depende apenas do id do agendamento, não do animal.
O script completo com CREATE TABLE, PKs, FKs, ENUMs e dados iniciais está em database.sql. Execute no MySQL antes de iniciar o servidor.

5. Padrões de Design

Factory factory.js

A classe EntidadeFactory centraliza a criação de todos os objetos do sistema. Em vez de criar objetos manualmente em vários pontos do código, todos passam por EntidadeFactory.criar<Tipo>(). Isso garante estrutura consistente e facilita alterações futuras.

Objetos criados: Usuário, Administrador, Animal, Ocorrência, Agendamento.

Singleton singleton.js

A classe BancoDeDados garante uma única instância do gerenciador de dados em toda a aplicação (lado cliente). Qualquer módulo que precise de dados usa BancoDeDados.getInstance(), evitando múltiplas cópias desincronizadas.

Responsabilidades: persistência em localStorage, CRUD em memória, inicialização de dados padrão.

Observer observer.js

O GerenciadorEventos implementa o padrão pub/sub: qualquer parte do sistema pode se inscrever em eventos (inscrever) e será notificada automaticamente quando ocorrerem (notificar).

Eventos monitorados: nova ocorrência, novo usuário, novo agendamento, atualização de status.

Resultado visual: sistema de toasts (notificações) disparado automaticamente a cada mudança de estado.

6. Deploy e Hospedagem

O frontend estático (pasta page/) está configurado para hospedagem na Vercel via vercel.json. Para o backend com MySQL, recomenda-se utilizar Railway ou Render, que suportam aplicações Node.js com banco de dados relacional.

Passos para deploy do frontend (Vercel)

  1. Faça push do repositório para o GitHub;
  2. Importe o projeto em vercel.com;
  3. O arquivo vercel.json já configura o roteamento para a pasta page/;
  4. Clique em Deploy.

Variáveis de Ambiente

Copie .env.example para .env e preencha com os dados reais. No Vercel/Railway, configure as variáveis diretamente no painel do serviço.
